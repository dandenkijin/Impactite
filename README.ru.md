# Impactite

🌐 [English](README.md) · **Русский** · [Deutsch](README.de.md)

Консольный (TUI) просмотрщик и редактор Markdown-файлов в стиле Obsidian,
написанный на Python с использованием библиотек **Rich** и **Textual**.

## Возможности

- 📁 **Древовидный список файлов** — навигация по файлам и папкам в левой панели
- 👁️ **Режим просмотра** — отрендеренный Markdown с форматированием и прокруткой
- ✏️ **Режим редактирования** — подсветка синтаксиса Markdown и кода в блоках
- 🏷️ **Облако тегов** — все теги с индивидуальными цветами, кликабельны
- 🔍 **Поиск по тегам** — модальное окно, результаты кликабельны
- 🎨 **Светлая/тёмная тема** — переключение на лету, выбор запоминается
- 🌐 **Локализация интерфейса** — английский, русский, немецкий
- 📝 **Формы ввода** — заметки-формы с полями и сохранением в файл или БД
- 🧮 **Псевдо-SQL запросы** — встраиваемые запросы к заметкам и БД (аналог Dataview)
- 🗂️ **Любая папка заметок** — каталог задаётся в конфиге (абсолютный/`~`/относительный)
- ⌨️ **Настраиваемые горячие клавиши**

## Требования

- Python 3.14+
- [uv](https://github.com/astral-sh/uv) (менеджер пакетов и окружений)

## Установка

```bash
uv sync
```

## Запуск

```bash
# Через CLI (после uv sync)
uv run impactite

# Или как модуль
uv run python -m impactite

# С указанием пути к конфигу
uv run impactite path/to/config.yaml
```

## Установка как исполняемой команды

Чтобы не запускать приложение каждый раз через `uv run`, можно установить его
как обычную команду `impactite` в системе. В проекте уже объявлена точка входа
(`[project.scripts]` в `pyproject.toml`), поэтому достаточно одной команды:

```bash
# Установить проект как глобальную команду (из корня проекта)
uv tool install .
```

После этого команда `impactite` доступна из любого каталога:

```bash
impactite                      # запуск
impactite path/to/config.yaml  # с указанием конфига
```

> **PATH.** `uv` размещает исполняемый файл в своём каталоге (обычно
> `~/.local/bin`). Если команда не находится, добавьте этот каталог в `PATH`:
> ```bash
> uv tool update-shell   # пропишет нужный каталог в PATH
> ```
> (после этого перезапустите терминал). Узнать путь можно через `uv tool dir`.

Полезные команды управления установленной командой:

```bash
uv tool install --editable .   # «editable»-установка: правки кода сразу видны,
                               # переустановка не нужна (удобно при разработке)
uv tool upgrade impactite      # обновить после изменений (если не editable)
uv tool list                   # показать установленные инструменты
uv tool uninstall impactite    # удалить команду
```

> **Важно про `config.yaml`.** После установки `impactite` запускается из любого
> каталога, а конфиг по умолчанию ищется в **текущем** каталоге. Передавайте путь
> к конфигу явно (`impactite ~/notes/config.yaml`) или запускайте из папки, где
> лежит `config.yaml`. Напомним: `notes_path` с относительным путём резолвится
> относительно самого конфиг-файла, так что каталог заметок не «съедет».

### Альтернатива: сборка дистрибутива (wheel)

Если нужен устанавливаемый пакет (например, чтобы поставить в другое окружение

uv build            # создаст dist/impactite-0.1.0-py3-none-any.whl и .tar.gz

## Конфигурация

Все настройки задаются в файле `config.yaml`. По умолчанию приложение ищет его
в текущем каталоге, но можно передать путь явно: `uv run impactite my-config.yaml`.

```yaml
# Путь к папке с markdown-файлами.
# Можно указать любой каталог:
#   - абсолютный путь:      "/home/user/Documents/MyNotes"
#   - с домашним каталогом: "~/Notes"
#   - относительный путь:   "./notes" (считается ОТНОСИТЕЛЬНО этого конфиг-файла)
notes_path: "./notes"

# Язык интерфейса: en (English), ru (Русский), de (Deutsch)
language: "ru"

# Горячие клавиши
hotkeys:
  open_file: "enter"          # Открыть выбранный файл
  edit_mode: "e"              # Переключить режим редактирования
  view_mode: "v"              # Переключить режим просмотра
  save_file: "ctrl+s"         # Сохранить файл
  search_tags: "ctrl+t"       # Поиск по тегам
  close_search: "escape"      # Закрыть поиск/диалог
  quit: "ctrl+q"              # Выход
  refresh: "ctrl+r"           # Обновить список файлов
  toggle_sidebar: "ctrl+b"    # Показать/скрыть боковую панель

# Настройки отображения
display:
  show_line_numbers: true     # Номера строк в редакторе
  word_wrap: true             # Перенос слов в редакторе
  syntax_theme: "monokai"     # Тема подсветки кода (monokai, dracula, github_light, ...)
  code_border: "round"        # Стиль рамки блоков кода (round, box, double)
  app_theme: "textual-dark"   # Тема приложения (см. ниже)

# Настройки облака тегов
tags:
  show_cloud: true            # Показывать облако тегов
  min_tag_size: 1             # Мин. размер тега
  max_tag_size: 3             # Макс. размер тега
```

### Параметры

| Параметр | Назначение |
|----------|------------|
| `notes_path` | Каталог с заметками. Относительный путь резолвится **относительно конфиг-файла**, а не текущей рабочей директории — запуск из любого места открывает один и тот же каталог. Поддерживается `~`. |
| `language` | Язык интерфейса: `en`, `ru`, `de`. Неизвестное значение → `en`. |
| `hotkeys.*` | Горячие клавиши для действий. |
| `display.show_line_numbers` | Номера строк в редакторе. |
| `display.word_wrap` | Перенос длинных строк в редакторе. |
| `display.syntax_theme` | Тема подсветки кода (Pygments-стили: `monokai`, `dracula`, `github_light`, …). |
| `display.code_border` | Стиль рамки вокруг блоков кода. |
| `display.app_theme` | Тема приложения. Сохраняется автоматически при переключении (`Ctrl+L`). |
| `tags.show_cloud` | Показывать ли облако тегов. |
| `tags.min_tag_size` / `max_tag_size` | Диапазон «веса» тегов в облаке. |

### Темы приложения (`app_theme`)

Тёмные: `textual-dark`, `dracula`, `monokai`, `nord`, `gruvbox`, `tokyo-night`, …
Светлые: `textual-light`, `solarized-light`, `catppuccin-latte`, `rose-pine-dawn`, `atom-one-light`.

`Ctrl+L` переключает светлую/тёмную тему прямо в приложении, а выбор
записывается обратно в `config.yaml` (поле `app_theme`).

## Теги

Теги распознаются двумя способами (в порядке приоритета):

1. **Во frontmatter** заметки — список `tags:`:
   ```yaml
   ---
   tags:
     - python
     - tutorial
   ---
   ```
2. **В тексте** заметки — в формате `#тег`:
   ```markdown
   Это заметка про #python и #textual.
   ```

При запуске все заметки сканируются, индекс тегов сохраняется в SQLite-базе
`.tag_index.db` внутри каталога заметок. Каждому тегу детерминированно
назначается уникальный цвет (сохраняется в БД). Облако тегов внизу слева и
теги прямо в тексте кликабельны — клик открывает поиск с этим тегом.

## Формы ввода

Любая заметка, у которой **первый ключ** frontmatter — `type: form`, в режиме
**просмотра** отображается как интерактивная форма ввода данных (с кнопками
«Сохранить» / «Отмена»). В режиме **редактирования** видна обычная разметка формы.

### Структура определения формы

```yaml
---
type: form                 # обязательно, должен быть ПЕРВЫМ ключом
destination: note          # куда сохранять: note (md-файл) или database (БД)
catalog: ./projects/       # подпапка (для note) или метка каталога (для database)
fields:
  - <ключ>: [<подпись>, <тип>, <параметр3>, <параметр4>]
  - ...
---
```

- **`type: form`** — маркер формы (обязательно первым).
- **`destination`** — `note` (по умолчанию) сохраняет результат как отдельный
  md-файл с frontmatter; `database` сохраняет запись в SQLite (та же БД, что и
  индекс тегов, таблица `form_records`).
- **`catalog`** — для `note` это подпапка внутри каталога заметок, куда ляжет
  созданный файл; для `database` — текстовая метка каталога записи.
- **`fields`** — список полей. Каждый элемент — словарь из одного ключа:
  `имя_поля: [подпись, тип, параметр3, параметр4]`.
  - `имя_поля` — ключ, под которым значение попадёт во frontmatter/БД (и по
    которому потом можно делать запросы).
  - `подпись` — текст метки над полем.
  - `тип` — тип поля (см. таблицу ниже).
  - `параметр3`, `параметр4` — зависят от типа.

### Типы полей

| Тип | Виджет | Параметр 3 | Параметр 4 | Сохраняемое значение |
|-----|--------|-----------|-----------|----------------------|
| `string` | однострочный ввод | макс. длина (число, `0` = без лимита) | — | строка |
| `text` | многострочный ввод | — | — | строка (многострочная) |
| `integer` | числовой ввод | — | — | целое число |
| `date` | ввод даты (плейсхолдер `ГГГГ-ММ-ДД`) | — | — | строка-дата |
| `boolean` | переключатель | — | — | `true` / `false` |
| `list` | список (см. ниже) | режим: `select` / `multi-select` | массив вариантов | строка или список |

Для `list` третий параметр задаёт поведение:

- **`select`** — выпадающий список, выбирается **один** вариант → сохраняется одна строка.
- **`multi-select`** — список с множественным выбором → сохраняется массив строк.
- иное/отсутствует — свободный ввод через запятую (`значение1, значение2 ...`),
  значение разбивается в массив строк.

Четвёртый параметр (`[вариант1, вариант2, ...]`) задаёт доступные варианты для
`select` / `multi-select`.

### Пример 1. Форма, сохраняемая в md-файлы

```yaml
---
type: form
destination: note
catalog: ./tasks/
fields:
  - name: ["Название задачи", string, 60]
  - date: ["Дата начала", date, 0]
  - enable: ["Активна", boolean, 0]
  - priority: ["Приоритет", list, select, [low, medium, high]]
  - tags: ["Теги", list, multi-select, [python, backend, ui, docs]]
  - count: ["Оценка (часы)", integer, 0]
  - description: ["Описание", text, 0]
---
```

При заполнении и сохранении создаётся файл вида `notes/tasks/Название_задачи.md`:

```yaml
---
name: Сделать парсер
date: '2026-06-02'
enable: true
priority: high
tags:
  - python
  - backend
count: 8
description: Распарсить входные данные и записать в БД
---
```

> Имя файла берётся из первого непустого строкового поля (очищается и
> подставляется), иначе — из метки времени.

### Пример 2. Форма, сохраняемая в базу данных

```yaml
---
type: form
destination: database
catalog: crm
fields:
  - title: ["Клиент", string, 80]
  - status: ["Статус", list, select, [new, active, closed]]
  - amount: ["Сумма", integer, 0]
  - notes: ["Заметки", text, 0]
---
```

Каждое сохранение добавляет строку в таблицу `form_records` (поля `id`,
`form_source`, `catalog`, `data` в JSON, `created_at`). Эти записи затем
доступны псевдо-SQL запросам через источник `database`.

## Псевдо-SQL запросы (аналог Dataview)

В режиме **просмотра** блок кода с языком `query` (или `dataview`) выполняется
как запрос и отображается в виде таблицы.

````markdown
```query
FROM notes
WHERE enable = true
SELECT name, count, category
ORDER BY count DESC
LIMIT 10
```
````

### Полный синтаксис

```
FROM notes|database[/<фильтр>]
WHERE <поле> <оператор> <значение> [AND ...]
GROUP BY <поле1>, <поле2>
HAVING <агрегат|поле> <оператор> <значение> [AND ...]
SELECT <поле1>, <поле2>, <агрегат> | *
ORDER BY <поле|агрегат> [ASC|DESC]
LIMIT <n>
```

Все секции, кроме `FROM`, необязательны. Регистр ключевых слов не важен.

### Источники данных (`FROM`)

- **`notes`** — строки из frontmatter md-файлов. Дополнительные неявные поля:
  `file` (имя файла без расширения) и `path` (путь относительно каталога заметок).
  Можно ограничить подпапкой: `FROM notes/tasks`.
- **`database`** — записи форм из таблицы `form_records`. Неявные поля:
  `id`, `catalog`, `source` (имя файла-формы), `created_at`. Можно ограничить
  каталогом: `FROM database/crm`.

### Операторы (`WHERE` / `HAVING`)

| Оператор | Значение |
|----------|----------|
| `=` | равно |
| `!=` | не равно |
| `>` `<` `>=` `<=` | числовое сравнение |
| `CONTAINS` | вхождение (для списков — есть ли элемент; для строк — подстрока) |
| `LIKE` | подстрока без учёта регистра |

Несколько условий объединяются через `AND`.

**Значения** распознаются автоматически: `true`/`false` (или `yes`/`no`) →
логическое, целое/дробное число → число, текст в кавычках → строка, иначе —
строка как есть. Примеры: `enable = true`, `count >= 10`, `category = разработка`,
`tags CONTAINS python`.

### Агрегатные функции и группировка

Доступны `COUNT`, `SUM`, `MIN`, `MAX`, `AVG`.

- `COUNT(*)` — число строк в группе; `COUNT(поле)` — число непустых значений.
- `SUM` / `AVG` — считаются только по числовым значениям (`AVG` округляется до 2 знаков).
- `MIN` / `MAX` — по числам, а если значения нечисловые (даты/строки) — лексикографически.

Агрегаты включаются, если есть `GROUP BY` **или** агрегат в `SELECT`. Без
`GROUP BY` агрегат считается по всем строкам (общие итоги). Если `SELECT`
опущен при `GROUP BY` — по умолчанию выводятся группирующие поля и `COUNT(*)`.

`HAVING` фильтрует уже сгруппированные строки. Можно фильтровать по агрегату,
которого **нет** в `SELECT` — он будет посчитан скрыто.

### Примеры запросов

**Простая выборка с фильтром и сортировкой:**

````markdown
```query
FROM notes/tasks
WHERE enable = true
SELECT name, count, category
ORDER BY count DESC
```
````

**Фильтр по тегу (вхождение в список):**

````markdown
```query
FROM notes/tasks
WHERE tags CONTAINS python
SELECT name, count, tags
```
````

**Сводка по категориям с агрегатами:**

````markdown
```query
FROM notes/tasks
GROUP BY category
SELECT category, COUNT(*), SUM(count), AVG(count), MIN(count), MAX(count)
ORDER BY SUM(count) DESC
```
````

**Общие итоги без группировки:**

````markdown
```query
FROM notes/tasks
WHERE enable = true
SELECT COUNT(*), SUM(count)
```
````

**Группировка + фильтрация по агрегату (`HAVING`):**

````markdown
```query
FROM notes/tasks
GROUP BY category
SELECT category, COUNT(*), SUM(count)
HAVING COUNT(*) > 1
ORDER BY SUM(count) DESC
```
````

**`HAVING` по агрегату, которого нет в `SELECT`:**

````markdown
```query
FROM notes/tasks
GROUP BY category
SELECT category, COUNT(*)
HAVING SUM(count) >= 20
```
````

**Запрос к записям форм в базе данных:**

````markdown
```query
FROM database/crm
WHERE status = active
SELECT title, amount, created_at
ORDER BY amount DESC
```
````

## Горячие клавиши (по умолчанию)

| Клавиша | Действие |
|---------|----------|
| `Enter` | Открыть выбранный файл |
| `E` | Режим редактирования |
| `Ctrl+S` | Сохранить файл |
| `Ctrl+T` | Поиск по тегам |
| `Ctrl+L` | Переключить светлую/тёмную тему |
| `Ctrl+R` | Обновить список файлов |
| `Ctrl+B` | Показать/скрыть боковую панель |
| `Ctrl+Q` | Выход |
| `Escape` | Закрыть поиск/диалог; выход из редактора (с запросом сохранения) |

В режиме просмотра доступна прокрутка: стрелки `↑`/`↓`, `PgUp`/`PgDown`,
`Home`/`End`. В модальных окнах фокус кнопок переключается стрелками `←`/`→`,
`Tab` и подтверждается `Enter`. Текст заметки можно выделять мышью для копирования
(в режимах просмотра и редактирования).
